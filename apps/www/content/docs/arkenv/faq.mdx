[//]: # "1. The Core Pitch & Proof"

## Why do I need ArkEnv?

We’ve seen teams struggle with environment variables in JavaScript projects. Because the runtime environment is an external dependency, it is notoriously brittle. Yet, we often avoid validating it because it feels like boilerplate. It’s tempting to assert with a quick `process.env.MY_VAR as string` and hope for the best.

But hope isn't an architecture. Variables go mysteriously missing during a CI/CD deployment. A boolean flags as `"false"` (a truthy string) and breaks your conditional logic. A teammate introduces a new variable without announcing it, leaving everyone else to chase down mysterious runtime errors.

We built ArkEnv to eliminate this exact class of failures. With a single initialization command (`npx @arkenv/cli@latest init`), you establish a **fail-safe** mechanism that guarantees your runtime configuration exactly mirrors your technical specifications before a single line of application code executes.

ArkEnv delivers native framework plugins, automated type coercion, and schema-aware AI agent support out-of-the-box. Read our full design philosophy in the [intro](/docs/arkenv) section.

## What does a failed validation look like?

With ArkEnv, your environment variables are **guaranteed to match your schema** at boot time. If a variable is missing, malformed, or fails validation, ArkEnv intercepts the process immediately, prevents the application from entering an unstable state, and outputs a clean, structured error to `stderr`:

```bash title="Terminal"
❯ PORT=hello npm start

ArkEnvError: Errors found while validating environment variables
  HOST must be a string or "localhost" (was missing)
  PORT must be a number (was a string)

```

## Does ArkEnv load my `.env` files?

**No.** ArkEnv validates your environment variables—it doesn't load them.

Most modern frameworks load `.env` files automatically. For plain Node.js, we recommend using [dotenv](https://www.npmjs.com/package/dotenv) or [tsx](https://tsx.is/).

ArkEnv intercepts the populated `process.env` or `import.meta.env` object, evaluates it against your schema, and applies [automatic type coercion](/docs/arkenv/coercion).

## Do I have to use ArkType?

**Not at all.** Any validator that implements the [Standard Schema](https://standardschema.dev) protocol is supported.

This includes:

- **Zod**
- **Valibot**
- **Typia**

See the [guide for details](/docs/arkenv/integrations/standard-schema).

## Does ArkEnv work with Zod?

Yes. **Zod is a first-class ArkEnv citizen**. We have an official [Zod example](https://github.com/yamcodes/arkenv/tree/main/examples/with-zod) in our repository and the [ArkEnv CLI](/docs/cli) can scaffold an ArkEnv + Zod configuration for you automatically.

## Why use ArkEnv instead of just raw ArkType or Zod?

Using "raw" ArkType or Zod for environment variables is tempting since it doesn't require any extra dependencies. However, it introduces security issues and boilerplate.

With ArkEnv:

- Sensitive, server-side variables fail loudly if used in client-side code so they don't leak.
- Diagnostics are optimized so you can spot failing variables clearly from the terminal.
- A variable typed `boolean` will be parsed as a `boolean`, not a `string` requiring manual handling.

[//]: # "3. Frameworks & Comparisons"

## Can I use ArkEnv with plain JavaScript?

Yes! While we recommend TypeScript for editor autocomplete and build-time safety, the validation engine works perfectly in plain JavaScript. Check out our [basic-js example](https://github.com/yamcodes/arkenv/tree/main/examples/basic-js).

## Is ArkEnv only for Next.js or Vite?

No. While we officially support Next.js, Node.js, Vite, and Bun through wrappers and plugins, the core API is based on standard JavaScript APIs and should run anywhere JavaScript does.

## How is ArkEnv different from T3 Env, or other similar tools?

[T3 Env](https://env.t3.gg/) is a well-established library that inspired much of our work. ArkEnv shares similar goals, but our underlying design choices differ:

- **ArkType is supported natively** for schema definitions, yielding extremely fast validation and cleaner schemas.
- **Built-in type coercion** and developer-friendly CLI tooling are included out of the box.
- **Vite and Bun are supported out of the box** alongside Next.js and Node.js.

Read more in the [comparison](/docs/arkenv/comparison) section.

[//]: # "4. Tooling, Extras & Trust"

## What does the ArkEnv CLI do?

The `@arkenv/cli` package streamlines configuration management. Instead of manually constructing config files, running the interactive initializer analyzes your workspace, detects your framework, lets you choose your validation primitive (ArkType, Zod, etc.), and generates a fully typed environment schema matching your environment.

Read the [CLI reference docs](/docs/cli) for a complete list of flags and capabilities.

## Does ArkEnv support AI development tools like Claude Code or Cursor?

Yes. ArkEnv is optimized for modern development workflows and features a dedicated [AI Agent Skill](/docs/arkenv/integrations/ai/skills). Providing this context to LLM-driven tools (like Claude Code, Cursor, Codex or Antigravity) allows your AI assistant to automatically maintain, update, and tighten your environment schemas as your application code evolves, so your schema changes match your code changes seamlessly.

## Is it safe to use in production?

Yes. ArkEnv is built to be a high-performance, low-overhead utility. The core runtime is tiny (\~2kB gzipped) and carries **zero external dependencies**, so it won't bloat your production dependency tree or introduce supply-chain vulnerabilities. It offloads the core parsing mechanics to your chosen, trusted validation library and safely shuts down the process if a configuration error is detected.

ArkEnv is currently in v0 and complies with Semantic Versioning 2.0.0. We're aggressively working towards v1 in a dedicated branch - track our progress on our [roadmap](https://github.com/yamcodes/arkenv/issues/683). The core validation engine is already used in production and not expected to change meaningfully.
