## Why do I need ArkEnv?

We’ve all seen teams struggle with environment variables in JavaScript projects. Because the runtime environment is an external dependency, it is notoriously brittle. Yet, we often avoid validating it because it feels like boilerplate. It’s tempting to assert with a quick `process.env.MY_VAR as string` and hope for the best.

But hope isn't an architecture. Variables go mysteriously missing during a CI/CD deployment. A boolean flags as `"false"` (a truthy string) and breaks your conditional logic. A teammate introduces a new variable without announcing it, leaving everyone else to chase runtime errors on Slack.

We built ArkEnv to eliminate this exact class of failures. With a single initialization command (`npx @arkenv/cli@latest init`), you establish a **fail-safe** mechanism that guarantees your runtime configuration exactly mirrors your technical specifications before a single line of application code executes.

ArkEnv delivers native framework plugins, automated type coercion, and schema-aware AI agent support out-of-the-box. Read our full design philosophy in the [intro](/docs/arkenv) section.

## What does a failed validation look like?

With ArkEnv, your environment variables are **guaranteed to match your schema** at boot time. If a variable is missing, malformed, or fails validation, ArkEnv intercepts the process immediately, prevents the application from entering an unstable state, and outputs a clean, structured error to `stderr`:

```bash title="Terminal"
❯ PORT=hello npm start

ArkEnvError: Errors found while validating environment variables
  HOST must be a string or "localhost" (was missing)
  PORT must be a number (was a string)

```

## Does ArkEnv load my `.env` files?

**No.** ArkEnv is a strict **validator**, not an environment **loader**.

Modern runtimes and meta-frameworks (Vite, Bun, Next.js, Remix) already feature native, highly optimized `.env` loading behavior. If you are running a bare Node.js application, we recommend pairing ArkEnv with a dedicated utility like [dotenv](https://www.npmjs.com/package/dotenv) or [tsx](https://tsx.hirok.io/).

ArkEnv intercepts the populated `process.env` (or a custom configuration object), evaluates it against your schema, and applies [automatic type coercion](/docs/arkenv/coercion).

## Do I have to use ArkType?

Not at all. While ArkEnv features native, zero-config integration with [ArkType](https://arktype.io/) (which we highly recommend for its performance and syntax), our core engine is entirely runtime-agnostic.

ArkEnv is built on top of the **Standard Schema** specification. This means you can drop in any compliant validator seamlessly:

- **Zod**
- **Valibot**
- **Typia**
- ...or any other library implementing the [Standard Schema](https://standardschema.dev) protocol.

For architectural implementation details, see [arkenv/standard](/docs/arkenv/standard).

## Does ArkEnv work with Zod?

Yes, natively. Because Zod implements the Standard Schema interface, ArkEnv treats it as a first-class citizen.

You can view our reference implementation in the [Zod example repository](https://github.com/yamcodes/arkenv/tree/main/examples/with-zod) or simply let the [ArkEnv CLI](/docs/cli) scaffold an ArkEnv + Zod configuration for you automatically.

## Why use ArkEnv instead of just raw Zod?

Writing custom validation wrappers around raw Zod schemas for every project introduces unnecessary, repetitive boilerplate:

1. **Automatic Coercion:** Environment variables are inherently read as strings by the OS. Manually chaining `z.coerce.number()` or parsing boolean strings (`"true"`/`"false"`) across dozens of variables is tedious. ArkEnv handles environmental type coercion automatically based on your target types.
2. **Terminal-Optimized Diagnostics:** Zod's default error structures are deeply nested and optimized for JSON APIs. ArkEnv intercepts these errors and formats them explicitly for terminal readability so engineers can diagnose deployment failures instantly.
3. **Framework Bundling Nuances:** In environments like Vite, client-side variables are statically replaced at build time via string substitution. ArkEnv’s dedicated plugins abstract this complexity away, ensuring compile-time and runtime compliance without manual hacking.

## Can I use ArkEnv with plain JavaScript?

Yes. While we highly recommend TypeScript to unlock compile-time typesafety, autocomplete, and seamless schema-to-editor type inference, the runtime validation engine works perfectly in standard JavaScript. Take a look at our [basic-js example](https://github.com/yamcodes/arkenv/tree/main/examples/basic-js) to understand the setup.

## Is ArkEnv only for Vite and Bun?

No. ArkEnv is architected to be completely runtime-agnostic. We maintain official, **battle-tested** integration suites for:

- **Node.js** (LTS and current releases)
- **Bun**
- **Vite**

Because the core library targets standard JavaScript APIs, ArkEnv is highly portable and runs smoothly on Deno or Edge environments (like Cloudflare Workers). However, if you are targeting specialized edge platforms and encounter any friction, please open an issue on our [GitHub repository](https://github.com/yamcodes/arkenv/issues).

## Can I use ArkEnv with Next.js or Nuxt?

Not officially.

While the core `arkenv` validation package can be manually integrated into these runtimes, meta-frameworks introduce highly specific client-side vs. server-side isolation rules that require distinct build-time abstractions. If you are building a dedicated Next.js or Nuxt application today, we highly recommend using [T3 Env](https://env.t3.gg).

We are actively researching isolated plugin architectures for meta-frameworks—track our progress on the [v1 roadmap](https://github.com/yamcodes/arkenv/issues/683).

## How is ArkEnv different from T3 Env?

[T3 Env](https://env.t3.gg/) is an exceptional tool and served as a core inspiration for ArkEnv. The primary differences lie in architectural focus and scope:

- If you are building inside the Next.js or Nuxt ecosystems, **T3 Env** remains the gold standard.
- If your stack relies on **Vite, Node.js microservices, Bun runtimes, standalone CLI tools, or Solid Start**, ArkEnv offers a more tailored, plugin-driven developer experience with deep ecosystem hooks.

Review our explicit architectural matrix in the [detailed comparison](/docs/arkenv/comparison).

## What does the ArkEnv CLI do?

The `@arkenv/cli` package streamlines configuration management. Instead of manually constructing config files, running the interactive initializer analyzes your workspace, detects your framework, lets you choose your validation primitive (ArkType, Zod, etc.), and generates a fully typed environment schema matching your environment.

Read the [CLI reference docs](/docs/cli) for a complete list of flags and capabilities.

## Does ArkEnv support AI development tools like Claude Code or Cursor?

Yes. ArkEnv is uniquely built for modern development workflows and features a dedicated [AI Agent Skill](/docs/arkenv/integrations/ai/skills). Providing this context to LLM-driven tools (like Claude Code, Cursor, or Windsurf) allows your AI assistant to automatically maintain, update, and tighten your environment schemas as your application code evolves, ensuring your schema changes match your code changes seamlessly.

## Is it safe to use in production?

Yes. ArkEnv is built to be a high-performance, low-overhead utility. The core runtime is tiny (\~2kB gzipped) and carries **zero external dependencies**, ensuring it won't bloat your production dependency tree or introduce supply-chain vulnerabilities. It offloads the core parsing mechanics to your chosen, trusted validation library and safely shuts down the process if a configuration error is detected.

While the project is currently in a stable `v0.y.z` release cycle as we finalize edge-case integrations (see our [v1 roadmap](https://github.com/yamcodes/arkenv/issues/683)), the core validation engine is **hardened**, thoroughly covered by our test suite, and actively relied upon in production environments.
