In this setup, you define your client, server, and shared schemas in a single file.

```files
env.ts
```

To protect server-side secrets from ever reaching the client bundle, ArkEnv uses a proxy that throws a runtime error if a server-side variable is detected.

:::warning
Defining your schemas together means the *server validation logic* is shipped in the client bundle.

| ⚠️ Visible in JS bundle                                                         | 🔒 Secure on server                                  |
| :------------------------------------------------------------------------------ | :--------------------------------------------------- |
| **Schema** (keys, types, and constraints)<br />e.g., `DATABASE_URL: string.url` | **Runtime values**<br />e.g., `"postgresql://db..."` |

If exposing your server variable names, types, or constraints is a security concern, use the [strict layout](/docs/nuxt/layouts/strict) instead.
:::

## Setup

The easiest way to bootstrap the simple layout is with the [ArkEnv CLI](/docs/cli). It automatically configures `@arkenv/nuxt` for your existing Nuxt project and generates your `env.ts` file.

<Tabs items={['npm', 'pnpm', 'yarn', 'bun']}>
  <Tab value="npm">
    ```bash
    npx @arkenv/cli@latest init --simple
    ```
  </Tab>

  <Tab value="pnpm">
    ```bash
    pnpm dlx @arkenv/cli@latest init --simple
    ```
  </Tab>

  <Tab value="yarn">
    ```bash
    yarn dlx @arkenv/cli@latest init --simple
    ```
  </Tab>

  <Tab value="bun">
    ```bash
    bunx @arkenv/cli@latest init --simple
    ```
  </Tab>
</Tabs>

## Your schema

When bootstrapping with the CLI, it generates a single `env.ts` file where you define your client, server, and shared schemas:

:::note
By default, ArkEnv for Nuxt uses an automatic code generation module (`@arkenv/nuxt/module`) in your Nuxt config to build the `runtimeEnv` block at compile time. This is why the schema file imports from `./generated/env.gen` rather than `@arkenv/nuxt` directly.
:::

```ts twoslash title="env.ts"
// @filename: ./generated/env.gen.ts
export { default } from '@arkenv/nuxt';
// @filename: env.ts
// ---cut---
import arkenv from "./generated/env.gen";

export const env = arkenv({
	server: {
		DATABASE_URL: "string",
	},
	client: {
		NUXT_PUBLIC_API_URL: "string",
	},
	shared: {
		NODE_ENV: "string",
	},
});
```

<Accordions>
  <Accordion title="Manual Setup">
    <Steps>
      <Step>
        ### Configure environment

        Create an `env.ts` file at the root of your project to define your schemas. Import `arkenv` from the auto-generated `./generated/env.gen.ts` helper file:

        ```ts twoslash title="env.ts"
        // @filename: ./generated/env.gen.ts
        export { default } from '@arkenv/nuxt';
        // @filename: env.ts
        // ---cut---
        import arkenv from "./generated/env.gen";

        export const env = arkenv({
        	server: {
        		DATABASE_URL: "string",
        	},
        	client: {
        		NUXT_PUBLIC_API_URL: "string",
        	},
        	shared: {
        		NODE_ENV: "string",
        	},
        });
        ```
      </Step>

      <Step>
        ### Register module in Nuxt config

        Add `@arkenv/nuxt/module` to your modules array in `nuxt.config.ts`. This statically scans your schema, generates the `env.gen.ts` file, registers environment variables to Nuxt's `runtimeConfig`, and automatically regenerates on changes:

        ```ts title="nuxt.config.ts"
        export default defineNuxtConfig({
        	modules: ["@arkenv/nuxt/module"]
        });
        ```

        You can customize the schema and output paths using optional parameters:

        ```ts title="nuxt.config.ts"
        export default defineNuxtConfig({
        	modules: ["@arkenv/nuxt/module"],
        	arkenv: {
        		schemaPath: "env.ts",              // Path to your schema (default: env.ts or src/env.ts)
        		outputPath: "generated/env.gen.ts"  // Custom output path
        	}
        });
        ```

        :::tip
        We recommend committing the generated `generated/env.gen.ts` file to Git. This ensures TypeScript types are immediately available on a clean clone or inside CI pipelines.
        :::
      </Step>

      <Step>
        ### Use in your code

        Import `env` throughout your Nuxt application:

        ```ts twoslash title="components/MyComponent.vue"
        // @filename: ./generated/env.gen.ts
        export { default } from '@arkenv/nuxt';
        // @filename: env.ts
        import arkenv from "./generated/env.gen";
        export const env = arkenv({
        	server: { DATABASE_URL: "string" },
        	client: { NUXT_PUBLIC_API_URL: "string" },
        	shared: { NODE_ENV: "string" },
        });
        // @filename: components/MyComponent.vue.ts
        // ---cut---
        import { env } from "../env";

        // Access is fully typesafe and autocompleted
        const apiUrl = env.NUXT_PUBLIC_API_URL;
        ```

        If you accidentally attempt to access a server-side secret in client-side code, it will throw a descriptive runtime exception during browser execution:

        ```ts twoslash title="components/ClientOnlyComponent.vue"
        // @filename: ./generated/env.gen.ts
        export { default } from '@arkenv/nuxt';
        // @filename: env.ts
        import arkenv from "./generated/env.gen";
        export const env = arkenv({
        	server: { DATABASE_URL: "string" },
        	client: { NUXT_PUBLIC_API_URL: "string" },
        	shared: { NODE_ENV: "string" },
        });
        // @filename: components/ClientOnlyComponent.vue.ts
        // ---cut---
        import { env } from "../env";

        // Throws a runtime error if evaluated in client-side code
        const dbUrl = env.DATABASE_URL;
        ```
      </Step>
    </Steps>
  </Accordion>
