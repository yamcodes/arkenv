In this setup, you define your client, server, and shared schemas in a single file.

```files
env.ts
```

To protect server-side secrets from ever reaching the client bundle, ArkEnv uses a proxy that throws a runtime error if a server-side variable is detected.

:::warning[Server logic in client bundle]
Defining your schemas together means the *server validation logic* is shipped in the client bundle.

| ⚠️ Visible in JS bundle                                                         | 🔒 Secure on server                                  |
| :------------------------------------------------------------------------------ | :--------------------------------------------------- |
| **Schema** (keys, types, and constraints)<br />e.g., `DATABASE_URL: string.url` | **Runtime values**<br />e.g., `"postgresql://db..."` |

The values themselves are *not* shipped to the client! But if exposing your server variable *names*, *types*, or *constraints* is a security concern, use the [strict layout](/docs/nuxt/layouts/strict) instead.
:::

## Setup

The easiest way to bootstrap the simple layout is with the [ArkEnv CLI](/docs/cli). It automatically configures `@arkenv/nuxt` for your existing Nuxt project and generates your `env.ts` file.

```npm
npx arkenv@latest init --simple
```

## Your schema

When bootstrapping with the CLI, it generates a single `env.ts` file where you define your client, server, and shared schemas:

```ts twoslash title="env.ts"
import arkenv from '@arkenv/nuxt';
// ---cut---
export const env = arkenv({
	server: {
		DATABASE_URL: "string",
	},
	client: {
		NUXT_PUBLIC_API_URL: "string",
	},
	shared: {
		NODE_ENV: "string",
	},
});
```

<Accordions>
  <Accordion title="Manual setup">
    ### Configure environment [step] [!toc]

    Create an `env.ts` file at the root of your project to define your schemas. Import `@arkenv/core` from `@arkenv/nuxt`:

    ```ts twoslash title="env.ts"
    import arkenv from '@arkenv/nuxt';
    // ---cut---
    export const env = arkenv({
    	server: {
    		DATABASE_URL: "string",
    	},
    	client: {
    		NUXT_PUBLIC_API_URL: "string",
    	},
    	shared: {
    		NODE_ENV: "string",
    	},
    });
    ```

    ### Register module in Nuxt config [step] [!toc]

    Add `@arkenv/nuxt/module` to your modules array in `nuxt.config.ts`. This statically scans your schema and registers environment variables to Nuxt's `runtimeConfig`:

    ```ts title="nuxt.config.ts"
    export default defineNuxtConfig({
    	modules: ["@arkenv/nuxt/module"]
    });
    ```

    You can customize the schema path using optional parameters:

    ```ts title="nuxt.config.ts"
    export default defineNuxtConfig({
    	modules: ["@arkenv/nuxt/module"],
    	arkenv: {
    		schemaPath: "env.ts"  // Path to your schema (default: env.ts or src/env.ts)
    	}
    });
    ```

    ### Use in your code [step] [!toc]

    Import `env` throughout your Nuxt application:

    ```ts title="components/MyComponent.vue"
    import { env } from "~~/env";

    // Access is fully typesafe and autocompleted
    const apiUrl = env.NUXT_PUBLIC_API_URL;
    ```

    If you accidentally attempt to access a server-side secret in client-side code, it will throw a descriptive runtime exception during browser execution:

    ```ts title="components/ClientOnlyComponent.vue"
    import { env } from "~~/env";

    // Throws a runtime error if evaluated in client-side code
    const dbUrl = env.DATABASE_URL;
    ```
