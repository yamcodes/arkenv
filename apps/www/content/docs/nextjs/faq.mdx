## I was able to access a client-side variable in a server context! Is that ok?

**Yes.** In Next.js, environment variables prefixed with `NEXT_PUBLIC_` are public and intended to be accessible everywhere.

`@arkenv/nextjs` permits accessing client variables in server-side contexts (like Server Components or API routes) by design. The runtime security proxy only blocks the reverse: accessing server-only variables (those without the `NEXT_PUBLIC_` prefix) on the client.

## Why do I need to map client variables in `runtimeEnv` but not server variables?

Next.js statically analyzes and inlines environment variables prefixed with `NEXT_PUBLIC_` at build time. To do this, the bundler looks for exact string matches like `process.env.NEXT_PUBLIC_VAR`.

If you access them dynamically via a validated object like `env.NEXT_PUBLIC_VAR`, the bundler cannot statically analyze it. By passing them into the `runtimeEnv` object (e.g., `NEXT_PUBLIC_VAR: process.env.NEXT_PUBLIC_VAR`), you give the Next.js compiler a static reference it can inline, which ArkEnv then reads.

Server-side variables, on the other hand, do not need to be inlined and are read dynamically from Node's runtime environment, so ArkEnv can fall back to reading them directly from `process.env` without any manual mapping in `runtimeEnv`.

## How does the runtime protection work under the hood?

`@arkenv/nextjs` automatically selects the correct environment logic using Node's package conditional exports. The integration ships with **two entry points** chosen by the Next.js bundler:

| Condition                                | Entry point            | `isServer` |
| ---------------------------------------- | ---------------------- | ---------- |
| `react-server` (RSC / Server Components) | `dist/react-server.js` | `true`     |
| `default` (Client Components + SSR)      | `dist/index.js`        | `false`    |

You always write `import arkenv from "@arkenv/nextjs"`—the bundler silently resolves the correct file.

The `isServer` flag is the mechanism behind runtime protection:

- **`true` (RSC)**: The full schema (`server + client + shared`) is validated, and server variables are read directly from `process.env`.
- **`false` (Client/SSR)**: Only the `client + shared` schema is validated. Accessing a key from the `server` schema throws a descriptive error during SSR pre-rendering and in the browser, ensuring sensitive data is never leaked.

This approach statically hardcodes the execution context at the bundler layer, which is more reliable than runtime heuristics like checking `typeof window`.

## What is the "Separate Files" mode?

By default, a single `env.ts` file works for both server and client variables because the default entry point (`@arkenv/nextjs`) is resolved conditionally by the bundler. Some teams prefer stricter separation—keeping server secrets in a dedicated file that can never be imported by client code.

The **Separate Files** mode provides exactly that:

- **`@arkenv/nextjs/server`** — only accepts `server` and `shared` schemas. It imports `server-only` at the top level, so the Next.js bundler will **fail the build** if any client component imports this file.
- **`@arkenv/nextjs/client`** — only accepts `client` and `shared` schemas. It can be safely imported anywhere.

Both entry points also expose a **`react-server`** sub-condition for RSC usage, mirroring the dual-bundler behavior of the default entry point.

### Why use separate files?

1. **Compile-time isolation** — `server-only` gives you a bundler-enforced guarantee. Even if you accidentally import the server file in a Client Component, the build fails immediately.
2. **Cleaner separation of concerns** — Server and client configuration live in distinct files (`env/server.ts` vs. `env/client.ts`), making it obvious which variables belong where.
3. **Native `extends` API** — Because the server file cannot directly import the client file in all architectures, we provide an `extends` option that merges validated outputs of other ArkEnv proxies while preserving proxy-level protections (unknown-key / typo checks and server-only access restrictions on the client).
