ArkEnv for Next.js supports any validator that implements the [Standard Schema](https://standardschema.dev/) specification — including Zod, Valibot, Typia, and others listed here:

  <Card title="Which schema libraries implement the spec?" href="https://standardschema.dev/schema#what-schema-libraries-implement-the-spec" description="Standard Schema" />
</Cards>

## Standard Mode (no ArkType)

Use `@arkenv/standard` and the `@arkenv/nextjs/standard` subpaths when you want Zod, Valibot, or another Standard Schema validator **without** installing `arktype`.

### Install

```package-install
@arkenv/standard @arkenv/nextjs zod
```

### Configure with codegen (recommended)

Wrap your Next.js config with `withArkEnv` from `@arkenv/nextjs/standard/config` so the generated helper imports the Standard Schema engine:

```ts title="next.config.ts"
import { withArkEnv } from "@arkenv/nextjs/standard/config";
import type { NextConfig } from "next";

const nextConfig: NextConfig = {};

export default withArkEnv(nextConfig);
```

Define a flat schema and import `arkenv` from the generated helper — no manual `runtimeEnv` mapping:

```ts title="src/env.ts" twoslash
// @filename: generated/env.gen.ts
export default function arkenv<const TSchema extends Record<string, unknown>>(
	schema: TSchema,
	options?: any,
): Readonly<TSchema> {
	return {} as any;
}

// @filename: env.ts
// ---cut---
import arkenv from "./generated/env.gen";
import { z } from "zod";

export const env = arkenv({
	DATABASE_URL: z.string().url(),
	NEXT_PUBLIC_API_URL: z.string().url(),
	NODE_ENV: z.enum(["development", "production", "test"]).default("development"),
});
```

:::tip
Prefer the [ArkEnv CLI](/docs/cli) (`npx arkenv@latest init`) and pick Zod or Valibot — it scaffolds Standard Mode for you.
:::

### Without codegen

If you disable codegen (`withArkEnv(nextConfig, { codegen: false })`), import from `@arkenv/nextjs/standard` and map client/`NODE_ENV` keys in `runtimeEnv` yourself:

```ts title="src/env.ts" twoslash
import arkenv from "@arkenv/nextjs/standard";
import { z } from "zod";

export const env = arkenv(
	{
		DATABASE_URL: z.string().url(),
		NEXT_PUBLIC_API_URL: z.string().url(),
		NODE_ENV: z.enum(["development", "production", "test"]).default("development"),
	},
	{
		runtimeEnv: {
			NEXT_PUBLIC_API_URL: process.env.NEXT_PUBLIC_API_URL,
			NODE_ENV: process.env.NODE_ENV,
		},
	},
);
```

See [Can I disable code generation?](/docs/nextjs/faq#can-i-disable-code-generation) for details.

:::note
Legacy nested layout `arkenv({ server, client, shared, runtimeEnv })` is still supported but deprecated. Prefer the [flat layout](/docs/nextjs/layouts/flat).
:::

## Mixing with ArkType

Already on `@arkenv/core`? You can mix ArkType DSL strings with Zod (or any Standard Schema validator) in the same schema — ArkType is still required as a peer dependency:

```ts title="src/env.ts" twoslash
// @filename: generated/env.gen.ts
import type { type as at, distill } from "arktype";
export default function arkenv<const T extends {}>(
	schema: T,
	options?: any,
): Readonly<distill.Out<at.infer<T>>> {
	return {} as any;
}

// @filename: env.ts
// ---cut---
import arkenv from "./generated/env.gen";
import { z } from "zod";

export const env = arkenv({
	HOST: "string.host",
	PORT: "number.port",
	DATABASE_URL: z.string().url(),
	NEXT_PUBLIC_API_URL: z.string().url(),
});
```

For the ArkType-free path, stay on [Standard Mode](#standard-mode-no-arktype) above. For core (non-Next) mixing examples, see [Standard Schema validators](/docs/arkenv/integrations/standard-schema).
