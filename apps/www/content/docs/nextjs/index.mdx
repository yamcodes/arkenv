> [!IMPORTANT]
> This integration requires [ArkType](https://arktype.io) to be installed.

The `@arkenv/nextjs` integration provides end-to-end environment validation for both the server and the browser. It catches configuration bugs early in your build pipeline and prevents sensitive server-side variables from ever reaching client components.

If a server secret is accidentally accessed in client code, the integration immediately fails your build or throws a descriptive runtime error. We leverage native Next.js conditional exports to guarantee this protection statically—see the [FAQ](/docs/nextjs/faq#how-does-the-runtime-protection-work-under-the-hood) for under-the-hood details.

## Quickstart

The easiest way to get started is with the [ArkEnv CLI](/docs/cli). It automatically configures `@arkenv/nextjs` for your existing Next.js project, or spins up a new one from a template.

<Tabs items={['npm', 'pnpm', 'yarn', 'bun']}>
  <Tab value="npm">
    ```bash
    npx @arkenv/cli@latest init
    ```
  </Tab>

  <Tab value="pnpm">
    ```bash
    pnx @arkenv/cli@latest init
    ```
  </Tab>

  <Tab value="yarn">
    ```bash
    yarn dlx @arkenv/cli@latest init
    ```
  </Tab>

  <Tab value="bun">
    ```bash
    bunx @arkenv/cli@latest init
    ```
  </Tab>
</Tabs>

<Accordions>
  <Accordion title="Manual Installation">
    <Steps>
      <Step>
        ### Installation

        ```package-install
        @arkenv/nextjs arktype
        ```
      </Step>

      <Step>
        ### Configure environment

        Create an `env.ts` file (typically in `src/env.ts` or at the root of your project) to define and validate your schemas:

        ```ts title="src/env.ts" twoslash
        import arkenv from "@arkenv/nextjs";

        export const env = arkenv({
        	server: {
        		DATABASE_URL: "string",
        	},
        	client: {
        		NEXT_PUBLIC_API_URL: "string",
        	},
        	shared: {
        		NODE_ENV: "string",
        	},
        	runtimeEnv: {
        		NEXT_PUBLIC_API_URL: process.env.NEXT_PUBLIC_API_URL,
        		NODE_ENV: process.env.NODE_ENV,
        	},
        });
        ```
      </Step>

      <Step>
        ### Use in your code

        Import `env` throughout your Next.js application:

        ```ts title="src/app/page.tsx" twoslash
        // @filename: env.ts
        import arkenv from "@arkenv/nextjs";
        export const env = arkenv({
        	server: { DATABASE_URL: "string" },
        	client: { NEXT_PUBLIC_API_URL: "string" },
        	runtimeEnv: {
        		NEXT_PUBLIC_API_URL: "https://api.example.com",
        	}
        });

        // @filename: page.tsx
        // ---cut---
        import { env } from "./env";

        // Access is fully typesafe and autocompleted
        const apiUrl = env.NEXT_PUBLIC_API_URL;
        //      ^?
        ```

        If you accidentally attempt to access a server-side secret in a Client Component, it will throw a descriptive runtime exception during both server-side pre-rendering (SSR) and browser runtime execution:

        ```ts title="src/app/client-component.tsx" twoslash
        // @filename: env.ts
        import arkenv from "@arkenv/nextjs";
        export const env = arkenv({
        	server: { DATABASE_URL: "string" },
        	client: { NEXT_PUBLIC_API_URL: "string" },
        	runtimeEnv: {
        		NEXT_PUBLIC_API_URL: "https://api.example.com",
        	}
        });

        // @filename: client-component.tsx
        // ---cut---
        import { env } from "./env";

        // Throws a runtime error if evaluated in a client component (during SSR or in the browser)
        const dbUrl = env.DATABASE_URL;
        //      ^?
        ```
      </Step>
    </Steps>
  </Accordion>
</Accordions>

## Separate Files Mode

If you prefer keeping server and client environment variables in separate files, you can use the dedicated `@arkenv/nextjs/server` and `@arkenv/nextjs/client` entry points. This mode uses the `server-only` package to provide compile-time guarantees that server secrets cannot be imported into client code.

<Steps>
  <Step>
    ### Server environment file

    Create a server-only environment file that imports `@arkenv/nextjs/server`:

    ```ts title="src/env/server.ts" twoslash
    import arkenv from "@arkenv/nextjs/server";

    export const serverEnv = arkenv({
      server: {
        DATABASE_URL: "string",
      },
      shared: {
        NODE_ENV: "string",
      },
      runtimeEnv: {
        NODE_ENV: process.env.NODE_ENV,
      },
    });
    ```
  </Step>

  <Step>
    ### Client environment file

    Create a client environment file that imports `@arkenv/nextjs/client`:

    ```ts title="src/env/client.ts" twoslash
    import arkenv from "@arkenv/nextjs/client";

    export const clientEnv = arkenv({
      client: {
        NEXT_PUBLIC_API_URL: "string",
      },
      shared: {
        NODE_ENV: "string",
      },
      runtimeEnv: {
        NEXT_PUBLIC_API_URL: process.env.NEXT_PUBLIC_API_URL,
        NODE_ENV: process.env.NODE_ENV,
      },
    });
    ```
  </Step>

  <Step>
    ### Combining with `extends`

    If your server code needs access to client variables, use the `extends` option to merge validated client outputs into the server environment:

    ```ts title="src/env/server.ts" twoslash
    // @filename: client.ts
    import arkenv from "@arkenv/nextjs/client";
    export const clientEnv = arkenv({
      client: { NEXT_PUBLIC_API_URL: "string" },
      runtimeEnv: { NEXT_PUBLIC_API_URL: "https://api.example.com" },
    });

    // @filename: server.ts
    // ---cut---
    import arkenv from "@arkenv/nextjs/server";
    // highlight-next-line
    import { clientEnv } from "./client";

    export const serverEnv = arkenv({
      server: {
        DATABASE_URL: "string",
      },
      shared: {
        NODE_ENV: "string",
      },
      // highlight-next-line
      extends: [clientEnv],
      runtimeEnv: {
        DATABASE_URL: process.env.DATABASE_URL,
        NODE_ENV: process.env.NODE_ENV,
      },
    });
    ```

    Accessing `serverEnv.NEXT_PUBLIC_API_URL` is now fully typed and safe. The underlying proxy still prevents leaking server-only variables to the client, even when extending.
  </Step>

  <Step>
    ### Shared definitions

    Both entry points accept a `shared` schema for variables used everywhere. Shared keys must be provided in both `runtimeEnv` objects when using separate files.
  </Step>
</Steps>

## Using Standard Schema validators (Zod, Valibot)

This integration uses ArkType for validation under the hood, which natively supports Standard Schema. You can freely mix ArkType DSL strings with Zod or Valibot validators in the same schema - no extra configuration needed.

Here is an example using Zod:

```ts title="src/env.ts" twoslash
import arkenv from "@arkenv/nextjs";
import { z } from "zod";

export const env = arkenv({
	server: {
		DATABASE_URL: z.string().url(),
	},
	client: {
		NEXT_PUBLIC_API_URL: z.string().url(),
	},
	shared: {
		NODE_ENV: z.enum(["development", "production", "test"]),
	},
	runtimeEnv: {
		NEXT_PUBLIC_API_URL: process.env.NEXT_PUBLIC_API_URL,
		NODE_ENV: process.env.NODE_ENV,
	},
});
```
